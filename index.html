Overview

exaPipeline is a production-ready, containerized system designed to transform domain-specific PDF documents into structured datasets for fine-tuning large language models (LLMs), specifically optimized for Qwen3:4B.

It automates the entire process from document ingestion to training data generation with high-quality control and minimal manual intervention.

Microservices Architecture

Scalable, fault-tolerant design with separate components for each processing stage.

AI-Powered Processing

Leverages advanced OCR and LLM technologies for accurate document analysis.

Real-time Monitoring

Track processing progress and system health through intuitive dashboards.

Purpose and Goals

exaPipeline bridges the gap between messy domain documents and AI training data. It achieves:

Full Automation

Transform PDFs into JSONL training data with minimal manual intervention.

Quality Assurance

Ensure high-quality data through validation and hallucination detection.

Scalable Architecture

Support large-scale processing with containerized microservices.

User-Friendly

Provide a no-code Streamlit dashboard for non-technical users.

Multiple Export Formats

Support SFT, RLAIF, RLHF, and Qwen3 native chat formats.

Deployment Instructions

Important: The pipeline uses several services with IP addresses. Never hard-code IPs in code. Always configure via .env files.

Prerequisites

  • Docker & Docker Compose
  • Git
  • exaOCR server (PDF → Markdown)
  • vLLM server running Qwen3:4B

Repository Links

  • exaOCR Repository

Configuration Files (Critical IPs)

All external service IPs must be configured in environment files:

# exaPipeline/.env
REDIS_URL=redis://<redis-host-ip>:6379/0          # e.g., 192.168.1.151
OCR_API_URL=http://<exaocr-ip>:45001             # e.g., 192.168.1.151:45001
VLLM_API_URL=http://<vllm-ip>:45000               # e.g., 192.168.1.151:45000

# exaPipelineDashboard/.env
BACKEND_API_URL=http://<backend-api-ip>:8000       # e.g., 192.168.1.151:8000

Single Machine Deployment

If you're running all services on the same machine, use localhost or 127.0.0.1 in your configuration:

# exaPipeline/.env
REDIS_URL=redis://localhost:6379/0
OCR_API_URL=http://localhost:45001
VLLM_API_URL=http://localhost:45000

# exaPipelineDashboard/.env
BACKEND_API_URL=http://localhost:8000
Tip: Using localhost is recommended for single-machine setups as it's more readable and less prone to configuration errors.

Multi-Machine Deployment

For production or distributed setups, use the actual IP addresses of each service:

# exaPipeline/.env
REDIS_URL=redis://192.168.1.151:6379/0          # Redis server IP
OCR_API_URL=http://192.168.1.152:45001          # exaOCR server IP
VLLM_API_URL=http://192.168.1.153:45000          # vLLM server IP

# exaPipelineDashboard/.env
BACKEND_API_URL=http://192.168.1.154:8000        # Backend API server IP
Warning: Ensure all machines can communicate over the network and that firewalls allow connections on the specified ports.
Warning: Some code files may contain default IPs (e.g., 192.168.1.151). These are placeholders. Always override them using .env files — Docker Compose loads these automatically.

Deployment Steps

Clone the Repositories

# Clone exaOCR
git clone https://github.com/ikantkode/exaOCR.git
cd exaOCR

# Clone exaPipeline
git clone https://github.com/ikantkode/exaPipeline.git
cd exaPipeline

# Clone exaPipelineDashboard
git clone https://github.com/ikantkode/exaPipelineDashboard.git
cd exaPipelineDashboard

Configure Environment Files

Edit the .env files in both exaPipeline and exaPipelineDashboard with your actual service IPs or localhost as shown above.

Deploy exaOCR (if needed)

cd exaOCR
docker compose up -d

This will start the exaOCR service on port 45001.

Deploy Backend Pipeline

cd exaPipeline
docker compose up -d

This starts 9 containers: API, Redis, 6 workers, and Flower monitoring.

Deploy Frontend Dashboard

cd exaPipelineDashboard
docker compose up -d

This starts the Streamlit dashboard on port 8501.

Verify Services

  • Dashboard: http://localhost:8501
  • API: http://localhost:8000/docs
  • Monitoring: http://localhost:5555

IP Configuration Guide

All network communication uses configurable IPs. Default placeholders (192.168.1.151) appear in some code files for development.

You MUST override these using .env files for production or multi-machine setups.

Required Services & Default Ports

Service Purpose Default Port Config Location
Redis Task queue 6379 exaPipeline/.env → REDIS_URL
exaOCR PDF → Markdown 45001 exaPipeline/.env → OCR_API_URL
vLLM (Qwen3) LLM inference 45000 exaPipeline/.env → VLLM_API_URL
Backend API Pipeline control 8000 exaPipelineDashboard/.env → BACKEND_API_URL
Dashboard User interface 8501 Local access

Network Configuration Examples

Localhost Configuration

For single-machine development, use localhost:

# exaPipeline/.env
REDIS_URL=redis://localhost:6379/0
OCR_API_URL=http://localhost:45001
VLLM_API_URL=http://localhost:45000

# exaPipelineDashboard/.env
BACKEND_API_URL=http://localhost:8000

Local Area Network Configuration

For distributed services on your local network:

# exaPipeline/.env
REDIS_URL=redis://192.168.1.100:6379/0
OCR_API_URL=http://192.168.1.101:45001
VLLM_API_URL=http://192.168.1.102:45000

# exaPipelineDashboard/.env
BACKEND_API_URL=http://192.168.1.103:8000

Docker Network Configuration

When using Docker networks, you can use service names:

# exaPipeline/.env
REDIS_URL=redis://redis-service:6379/0
OCR_API_URL=http://exaocr-service:45001
VLLM_API_URL=http://vllm-service:45000

# exaPipelineDashboard/.env
BACKEND_API_URL=http://backend-service:8000
Best Practice: Use host network mode or proper DNS in production. Never rely on hard-coded 192.168.1.151 in code.

vLLM Server Setup

Deploy vLLM with Qwen3:4B:

docker run --gpus all -p 45000:80 vllm/vllm-openai:latest \
  --model Qwen/Qwen3-4B

Then set VLLM_API_URL=http://your-server-ip:45000 in .env

Note: Ensure the vLLM server is running before starting the pipeline. Slow or unavailable vLLM servers cause annotation/synthesis delays.

Troubleshooting

Connection Errors

Verify all IPs in .env files match actual server addresses. Check network connectivity and firewall settings.

No Validated Data

Check validation worker logs, ensure filename formats match (e.g., chunk_0001_syn_01.json).

vLLM Errors

Ensure vLLM server is running and accessible. Restart with: docker restart vllm-container

Dashboard Issues

Check logs: docker logs exapipelinedashboard-streamlit-1. Verify BACKEND_API_URL is correct.
